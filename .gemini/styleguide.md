# Style guide for eth-protocol-fellowship

Gemini Code Assist applies this guide when reviewing pull requests in this repo.
It is the public site for Ivan Anishchuk's Ethereum Protocol Fellowship (EPF)
cohort-seven dev updates, built with [zensical](https://zensical.org) and
deployed to GitHub Pages. Most changes are Markdown content, `zensical.toml`
config, the GitHub Actions workflow, and one Python build script
(`tools/social_card.py`).

Gemini Code Assist is a best-effort second opinion. The review gate is
`/code-review`, the strict zensical build, and Ivan's self-review. A green or
empty Gemini review is not an approval.

## Hard rules (flag any violation, never suggest breaking these)

- **Dev-update prose is the fellow's own writing.** Posts under `updates/` are
  EPF weekly dev updates, and EPF requires the body prose to be the fellow's own
  work. Never suggest rewriting that prose or replacing it with generated text.
  Review only the front matter, links, and formatting of those files.
- **No third-party artwork committed.** Social cards are generated by our own
  build step from the theme colors, our logo, and the page title. Do not suggest
  committing Ethereum Foundation illustrations.
- **Public repo hygiene.** Flag any secret, large binary, or build output added
  to the tree. The generated `site/` and `docs/updates/` trees are gitignored.

## Writing style

Apply to all prose (docs, comments, commit messages, PR text):

- No em-dashes used as subphrase separators. Use commas, or split into two
  sentences.
- No contrastive negation. State the positive directly instead of "not X, but Y".
- Vary sentence length. Prefer active voice.
- No filler openers ("Let's dive in") or summary closers ("In conclusion").
- Avoid AI-tell vocabulary: delve, leverage, robust, seamless, tapestry,
  landscape, and similar.

## Front matter

Every Markdown page needs a `description:` line for its meta tag. Dev-update
posts also need `date`, `title`, and `categories`. Write `categories`/`tags` as
YAML block lists:

```yaml
categories:
  - weekly-update
```

Never as flow lists (`categories: [weekly-update]`): zensical's strict link
check reads `[x]` in front matter as an unresolved Markdown reference link and
fails the build. Allowed categories: `weekly-update`, `project-update`, `news`.
Flag a flow list. Do not report a block-list `[x]` that is not present.

## Build and CI

- The build is a strict gate: `zensical-updates build`, then
  `zensical build --clean --strict`, then `python tools/social_card.py`. Flag
  changes that would break it.
- Imports go at the top of any Python file (`tools/social_card.py`).
- Deliberate hardening, do not flag as issues: GitHub Actions pinned to full
  commit SHAs, top-level `permissions: {}` with per-job least-privilege grants,
  `persist-credentials: false`, and the harden-runner step.

Prefer a few high-confidence, high-severity findings over many style nits.
