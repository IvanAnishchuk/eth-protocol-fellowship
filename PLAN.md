# eth-protocol-fellowship, Build Plan

A polished, SEO and social-optimized public site for Ivan's EPF cohort-seven dev
updates and project updates. zensical, deployed to GitHub Pages, no custom domain.

Status: skeleton only. This repo currently holds `CLAUDE.md` and this plan. A
dedicated agent picks up implementation from Phase 0.

## Settled decisions

- **Renderer:** zensical. This is Ivan's explicit choice and his house pattern.
  zensical does not yet ship a blog plugin or social-card generation, so we add
  both ourselves by vendoring or extending a plugin and by writing a small build
  step. `mkdocs.yml`-style config compatibility is a side benefit, not the reason.
- **Content model:** a hub plus an "Updates" blog. Categories: `weekly-update`,
  `project-update`, `news`.
- **Repo:** public `IvanAnishchuk/eth-protocol-fellowship`, served at
  `https://ivananishchuk.github.io/eth-protocol-fellowship/`.
- **Social cards:** generated by our own build step from theme colors, our own
  logo, and the page title. Ethereum visuals inform palette and layout only; none
  are committed.
- **License:** CC0-1.0, in `LICENSE.md` (matches `cuere`).
- **Branding:** starter artwork in `docs/assets/` (an original hexagon, the SSZ
  container, wrapping a small Merkle tree, in an Ethereum-inspired indigo and
  violet gradient): `logo.svg`, `favicon.svg`, `social-card-bg.svg`. Placeholder
  quality; Ivan refines later.
- **Hub scope (v1):** link public artifacts only (proposal, repos, PRs). No
  private board link for now.

## Open items

- Palette refinement and the final logo or favicon. Starter art is in
  `docs/assets/`; Ivan refines later.
- Blog mechanism (decide in Phase 3): `knu2xs/zensical-blog` pinned, the same

## Phase 0, Repo and tooling

- `pyproject.toml` (uv): `zensical` (pin as in `cuere`), `zensical-blog` from a
  pinned git commit, plus the social-card generator's dependencies (Pillow or
  cairosvg).
- `.gitignore`: `site/`, `docs/.bloggen/`, `.venv/`, `__pycache__/`, `.cache/`.
- `README.md`: what the repo is, plus the `uv run zensical serve` preview steps.
- `LICENSE.md` (pending the license choice above).
- **Acceptance:** `uv run zensical build --clean --strict` passes on the empty
  scaffold.

## Phase 1, Theme, branding, SEO base

- `zensical.toml` (copy the structure from `cuere/zensical.toml`): `site_name`,
  `site_url` (canonical, also needed by the social cards), `site_description`,
  `repo_url`, `copyright`.
- Theme: an Ethereum-inspired palette with a dark and light toggle, a logo and
  favicon, the standard navigation and content features, and the Material
  markdown-extension block.
- SEO: per-page meta descriptions, canonical URLs from `site_url`. Verify whether
  zensical emits a sitemap; if it does not, generate one in the build step.
- **Acceptance:** a local build renders the hub; the `<head>` carries description
  and canonical tags.

## Phase 2, Hub pages

- `docs/index.md`: the hub. Who Ivan is, the etheorem formal-verification
  project, and links (proposal, repos, PRs, board if public).
- `docs/project.md`: the etheorem project overview, scope, and links.
- `docs/about.md`: a short bio, EPF context, and contact.
- Wire the nav: Home, Project, Updates, About.
- An agent may draft these pages; Ivan revises them into his voice.
- **Acceptance:** nav and pages build; a strict build finds no broken links.

## Phase 3, Updates blog

- Add the blog via `knu2xs/zensical-blog`, pinned, configured under
  `[project.extra.zensical_blog]` in `zensical.toml` (`blog_dir`, `posts_dir`,
  drafts, tags, categories, archive).
- Spike: confirm the plugin's build integration (a single `zensical build`, or a
  pre-build generate step that writes `docs/.bloggen/`). Wire the blog into the
  nav manually rather than with the plugin's config-mutating `--apply-nav`.
- `docs/blog/index.md`: the "Updates" landing. `docs/blog/posts/`: the posts.
- Empty Week-0 slot at `docs/blog/posts/2026-06-DD-week-0.md`: front matter
  (`date`, `categories: [weekly-update]`, tags) and the kickoff headings only.
  The body stays empty with a clear TODO. Ivan writes the prose (no-LLM).
- Fallback: if the plugin does not integrate cleanly, hand-roll a dated-posts
  index. The posts are plain markdown, so no content is lost in the swap.
- **Acceptance:** the blog index, the post, and the category and archive pages
  render.

## Phase 4, Social cards

- Build a generation step that, per page, renders an Open Graph card (logo,
  palette, title) and injects the `og:image` and `twitter:image` meta into the
  page head. Pick the mechanism in the spike above.
- Cache generated cards; commit none of them as source.
- Document any system libraries the generator needs for CI.
- **Acceptance:** the build emits per-page social images; pages carry `og:image`
  and `twitter:image` meta; a sample card looks right.

## Phase 5, CI and Pages deploy

- `.github/workflows/docs.yml` adapted from `cuere`: pull requests run a strict
  build as the gate; a push to `main` deploys.
- Add the social-card generation step and any system libraries it needs.
- `uv sync --frozen --no-dev`; `uv run --no-sync zensical build --clean
  --strict`; `upload-pages-artifact` with `site/`; `deploy-pages`.
- Pin action SHAs, add harden-runner, set least-privilege permissions. Gitignore
  `docs/.bloggen/`.
- One-time: create the public GitHub repo; set Settings, Pages, Source to GitHub
  Actions.
- **Acceptance:** CI is green; the site is live at the URL; a shared link shows a
  social card.

## Phase 6, Polish

- Footer social links.
